# supabase-posthog — Stacker Pipe Demo

This project demonstrates how to connect two applications with a Stacker **pipe**:

1. Deploy **Supabase** as the primary app.
2. Deploy **PostHog** as an additional service in the same project.
3. Create a pipe from Supabase to PostHog so events flow between them.

## Architecture

- **Supabase** (main `app`): Kong gateway on `8000/tcp` and `8443/tcp`.
- **PostHog** (service `posthog`): analytics app on `8001/tcp` (mapped from container port `8000`).
- Both apps share the same Docker network created by Stacker, so they can reach each other by service name.
- Kong routes `/posthog/` to the PostHog receiver, so events can flow through the API gateway.

## Prerequisites

- Stacker CLI is installed and authenticated (`stacker login`).
- A `.env` file exists in this directory (created automatically by `generate-secrets.sh`).
- For cloud/existing-server targets, a cloud credential or existing server is configured.

## Initial setup

```bash
cd stacker-projects/supabase-posthog

# Create .env and generate secrets
./scripts/generate-secrets.sh

# Load root credentials (cloud / existing-server keys)
set -a && source ../../.env && set +a
```

## Deployment flow

### 1. Deploy Supabase

Supabase is the primary application defined in `stacker.yml` under `app:`.

```bash
# Local
cd stacker-projects/supabase-posthog
stacker deploy --target local --force-rebuild --watch

# Cloud
stacker deploy --target cloud --key htz-0 --force-new --watch

# Existing server
stacker config setup server --ip "$EXISTING_SERVER_HOST" --user "$EXISTING_SERVER_USER" --key "$BASE_PATH/$EXISTING_SERVER_KEY"
stacker target server
stacker deploy --target server --force-rebuild --watch
```

Verify Supabase is healthy:

```bash
# For local
curl -I http://localhost:8000/

# For cloud / existing server
curl -I http://<SERVER_IP>:8000/
```

### 2. Deploy PostHog

PostHog is defined as a service in `stacker.yml`. After Supabase is running, deploy it with:

```bash
# Deploy the posthog service through the remote app deploy path
stacker service deploy posthog --force

# Alternative using the agent directly
stacker agent deploy-app posthog --force
```

Verify PostHog is healthy:

```bash
# Local
curl -I http://localhost:8001/

# Remote
curl -I http://<SERVER_IP>:8001/
```

### 3. Create the pipe

First, scan the running apps to see the endpoints Stacker can discover:

```bash
stacker pipe scan --capture-samples
```

Then create a pipe from Supabase to PostHog:

```bash
# Automatic endpoint matching
stacker pipe create supabase-posthog posthog

# Manual matching (skip AI/automatic discovery)
stacker pipe create supabase-posthog posthog --manual
```

Stacker will ask you to pick the source endpoint and the target endpoint. A typical real-world pipe maps:

- **Source**: Supabase `rest` service (`/rest/v1/<table>`) via the Kong gateway.
- **Target**: PostHog `/capture` endpoint (`POST` event capture API) at `/posthog/capture` through Kong, or directly at `http://posthog:8000/capture` on the internal network.

Activate the pipe:

```bash
# List pipes to get the instance ID
stacker pipe list

# Activate it
stacker pipe activate <PIPE_INSTANCE_ID>
```

Trigger it manually for a one-shot test:

```bash
stacker pipe trigger <PIPE_INSTANCE_ID>
```

### 4. Deploy the pipe to the remote server (optional)

If you created the pipe locally and want to promote it to a cloud/existing-server deployment:

```bash
stacker pipe deploy <PIPE_INSTANCE_ID> --deployment <DEPLOYMENT_HASH>
```

## Useful pipe commands

| Command | Purpose |
|---|---|
| `stacker pipe scan` | Discover endpoints on running apps |
| `stacker pipe create <source> <target>` | Create a new pipe |
| `stacker pipe list` | List active pipes |
| `stacker pipe activate <id>` | Start listening for triggers |
| `stacker pipe deactivate <id>` | Stop listening |
| `stacker pipe trigger <id>` | Run the pipe once manually |
| `stacker pipe history <id>` | Show execution history |
| `stacker pipe replay <id>` | Replay a previous execution |
| `stacker pipe deploy <id>` | Promote local pipe to remote deployment |

## Verification checklist

- [ ] Supabase responds on port `8000` (HTTP 200/302).
- [ ] PostHog responds on port `8001` (HTTP 200).
- [ ] `stacker pipe list` shows the created pipe.
- [ ] `stacker pipe trigger <id>` completes without errors.
- [ ] PostHog receives an event from Supabase.

## Notes

- PostHog container port `8000` is mapped to host port `8001` to avoid conflicting with Supabase Kong.
- Supabase Postgres is on `127.0.0.1:5432`.
- The `kong.yml` file is regenerated by `generate-secrets.sh` so the `ANON_KEY` and `SERVICE_ROLE_KEY` match the `.env` values.
